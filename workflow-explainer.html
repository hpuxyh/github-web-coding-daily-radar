GitHub 热榜流程图

小白版：这个系统每天帮你在 GitHub 上找新项目，先按规则筛一遍，再读项目说明，最后写成中文榜单。

核心原则：不能只看项目名和 stars。最终出现在榜单里的项目，必须读 README，再总结成人话。

第一步：搜索

先把可能有价值的新项目找出来。

搜哪些项目？

主要搜最近 90 天内出现、还处在早期阶段的项目。

↓
搜索方向
  • 各种有用、有趣的新应用和小工具
  • 各职业的效率优化：产品、运营、设计、销售、研究、写作、教育、财务等
  • 生活、学习、办公、创作、内容生产、知识管理、自动化工具
  • AI 工具、浏览器插件、桌面应用、低代码工具也会继续覆盖
↓
可选辅助信号

专家收藏或引用只作为辅助来源，默认不混入主榜。

第二步：筛选

从一堆候选里挑出更值得看的项目。

先看基础热度
  • stars 有没有增长
  • fork 有没有增长
  • 最近有没有更新
  • issue / 讨论是否活跃
↓
再看项目质量
  • README 是否说清楚
  • 有没有截图或 Demo
  • 有没有官网或使用入口
  • 项目是否还在维护
↓
最终入榜前必须读 README

只要它会展示在页面上，就重新读项目说明，避免把项目讲错。

第三步：展示

把技术项目翻译成普通人能看懂的榜单。

写成中文项目画像
  • 它是什么
  • 给谁用
  • 解决什么问题
  • 怎么解决
  • 省掉什么时间
↓
放进 4 个榜单

今日新增、大家都在用、早期潜力、参与讨论。

↓
生成当天页面

每天保存一个快照，页面可以按日期查看。

搜索逻辑和筛选逻辑

搜索逻辑

不是随机搜，而是围绕“最近可能冒出来的新工具”去搜。

最近 90 天 早期项目 有用应用 有趣工具 职业效率 办公自动化 内容创作 学习研究 生活工具 AI / 插件 / 桌面应用

筛选逻辑

不是 stars 越高越好，而是看它现在是不是正在变热、能不能看懂、能不能上手。

新增 stars fork 增长 最近更新 README 完整 有截图/Demo 可选专家关注 issue 活跃 早期潜力

之前为什么会错

之前有时只看标题、topics 和少量 README，就会猜错项目用途。比如项目其实是八字紫微工具，却被写成 AI 编程助手。

现在怎么避免

现在最终入榜前会强制读 README 和截图，再按项目真实内容分类。浏览器插件写成浏览器插件，知识库写成知识库，八字紫微工具写成八字紫微工具。

口径限制图

入什么项目会进候选池

  • 优先看最近 90 天创建的新项目。
  • 优先看 stars 还在 1000 以内的早期项目。
  • 方向要接近“有用或有趣的应用”：职业效率、生活工具、学习研究、内容创作、知识管理、办公自动化等都可以。
  • AI 工具、浏览器插件、知识库、写作工具、桌面应用、低代码只是例子，不是边界。
  • 专家收藏或引用只作为可选辅助来源，默认不混入主榜。

限什么项目会被限制

  • 太老、太成熟的大项目，不默认进“早期趋势”榜。
  • 超过 1000 stars 的项目，一般只做历史参考或辅助信号参考。
  • 没有 README、没有说明、看不出用途的项目，会降低排序。
  • 同一个项目不会反复出现在多个榜单里，会尽量只放到最合适的一个榜。

辅哪些信号只是辅助

  • stars 高不等于一定上榜，只是参考。
  • 可选辅助信号不是直接保送，只是加权参考。
  • 截图和 Demo 不决定排名，但能帮助判断能不能上手。
  • README 摘要不是照抄，而是读完后翻译成普通人能懂的话。
简单口径：新 + 早期 + 有用或有趣 + 对某类人能提效 + 最近有增长 + README 讲得清楚，才更容易进入榜单。

4 个榜单怎么理解

今日新增 今天增长快、刚冒出来的新项目。
大家都在用 fork、截图、说明更完整，看起来更能上手。
早期潜力 还不大，但方向有意思、增长不错。
参与讨论 issue、更新、协作比较活跃。
